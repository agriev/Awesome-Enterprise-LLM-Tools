# Contributing to Awesome LLM Tools for Enterprise

Thank you for your interest in contributing to this repository! This document provides guidelines and instructions for contributing.

## How to Contribute

### Reporting Issues

If you find a bug, have a suggestion, or want to request a new feature:

1. Check if the issue already exists
2. Create a new issue with a clear title and description
3. Include relevant details (version, environment, steps to reproduce)
4. Add appropriate labels if you have permission

### Adding Tools

To add a new tool to the repository:

1. **Fork the repository**
2. **Choose the right file**:

3. **Add your tool** following the existing format:
   - Description
   - Enterprise features
   - GitHub/Website link
   - Use cases
4. **Test your changes** (markdown formatting, links)
5. **Submit a pull request** with a clear description

### Adding Reference Architectures

To add a new reference architecture:

1. **Create a new file** in `reference-architectures/` directory
2. **Follow the template** structure:
   - Overview
   - Business Value
   - Architecture Components
   - Architecture Diagram
   - Implementation Details
   - Security Considerations
   - Example Use Cases
   - Monitoring & Observability
   - Integration Points
   - Performance Targets
   - Related Documents
   - Tools & Technologies
3. **Update the README** to include your architecture
4. **Submit a pull request**

### Improving Documentation

- Fix typos and grammar
- Improve clarity and structure
- Add missing information
- Update outdated content
- Translate content (if applicable)

## Contribution Guidelines

### Code of Conduct

- Be respectful and inclusive
- Provide constructive feedback
- Focus on what is best for the community
- Show empathy towards other community members

### Pull Request Process

1. **Fork the repository**
2. **Create a feature branch** (`git checkout -b feature/amazing-feature`)
3. **Make your changes**
4. **Commit your changes** (`git commit -m 'Add amazing feature'`)
5. **Push to the branch** (`git push origin feature/amazing-feature`)
6. **Open a Pull Request**

### Pull Request Guidelines

- **Clear Title**: Use a clear, descriptive title
- **Description**: Provide a detailed description of changes
- **Scope**: Keep changes focused and atomic
- **Testing**: Test your changes before submitting
- **Documentation**: Update documentation if needed

### Formatting Guidelines

#### Markdown Files

- Use proper heading hierarchy
- Format code blocks with language tags
- Use tables for structured data
- Include links where appropriate
- Follow existing style conventions

#### Tool Entries

When adding tools, use this format:

```markdown
### Tool Name
- **Description**: Brief description of the tool
- **Enterprise Features**: Key enterprise features
- **GitHub/Website**: Link to the tool
- **Use Cases**: Common use cases
```

#### Architecture Documents

- Use clear section headings
- Include architecture diagrams (Mermaid or images)
- Provide code examples where relevant
- Include security considerations
- Add monitoring and observability sections

## Review Process

1. **Automated Checks**: PRs are checked for formatting and links
2. **Maintainer Review**: A maintainer will review your PR
3. **Feedback**: You may receive feedback or requested changes
4. **Approval**: Once approved, your PR will be merged

## Questions?

If you have questions about contributing:

- Open an issue with the `question` label
- Check existing issues and discussions
- Review existing documentation

## Recognition

Contributors will be recognized in:
- Pull request acknowledgments
- Release notes (for significant contributions)
- Contributor list (for regular contributors)
